당신은 이 과제 저장소를 PROBLEM.md 요구사항에 맞게 수정해야 한다.

목표:
1) 서버 시작 시점에 초기 데이터가 자동으로 생성되어야 한다.
   - GET /health 가 200 OK 를 반환하는 시점에는 이미 데이터가 준비되어 있어야 한다.
   - 데이터 생성은 1분 이내에 완료되어야 한다.
2) 최소 데이터 규모를 만족해야 한다:
   - Department >= 10
   - Course >= 500
   - Student >= 10,000
   - Professor >= 100
3) 데이터는 정적 파일(SQL/CSV) 없이, 실행 시 프로그래밍 로직으로 동적 생성되어야 한다.
   - 다만 소규모 토큰 목록(이름/학과명 등)은 코드에 포함 가능.
   - "User1", "Course1" 같은 의미 없는 데이터는 피하고 현실적인 패턴을 사용한다.
4) 동시성 제어(정원 초과 방지)는 유지/강화한다.
   - 동시에 100명이 신청해도 남은 자리가 1이면 정확히 1명만 성공하도록 보장.
5) 문서도 PROBLEM.md에 맞게 갱신한다.
   - docs/REQUIREMENTS.md에서 데이터 규모 축소 방침을 제거하고, 실제 구현된 대규모 생성 방식/성능(1분 이내)과 정합성 보장 방식을 서술한다.
   - API 문서는 docs/ 디렉토리 아래로 이동하거나 복사하여 제출 구조를 준수한다(예: docs/API.md).

작업 지시:
- 먼저 현재 프로젝트에서 시드/초기화가 어떻게 이루어지는지 코드를 찾아라.
  (예: app startup hook, lifespan, db init 모듈, alembic/DDL, 기존 seed 함수 등)
- 앱 시작 시 자동 시드를 수행하도록 변경하라.
  - 이미 데이터가 존재하면 재생성하지 않거나(권장), 재생성 정책을 명확히 정하라.
- 데이터 생성 로직을 구현하라:
  - Department 10개 이상: 현실적인 학과명 토큰 목록 사용.
  - Professor 100명 이상: 한국식/영문식 이름 토큰 조합 등.
  - Student 10,000명 이상: 현실적인 이름 생성(토큰 조합) + max_credits=18 기본.
  - Course 500개 이상: 학과/교수/학기 배정, credits(예: 1~3) 분포, capacity(예: 10~60) 합리적 분포.
  - CourseTime 생성: 요일/시간대를 합리적으로 생성하되, 같은 시간대 중복 수강 불가 규칙을 테스트할 수 있도록 일부는 겹치게 생성해라.
- 성능(1분 이내)을 위해 bulk insert / transaction batching을 적용하라.
  - SQLite 사용 시 WAL 모드/적절한 인덱스 생성, executemany, 단일 트랜잭션 활용 등.
- 목록 조회 API의 응답은 최소 필드(정원/현재 인원/시간 정보)를 계속 포함하도록 유지하라.
- 테스트/검증:
  - /health 200 이후 /students, /courses 호출 시 최소 수량 이상이 반환되는지 확인하는 스모크 테스트(간단한 유닛/통합 테스트 또는 스크립트)를 추가하라.
  - 동시성 테스트(정원 1 남은 강좌에 동시 신청)도 유지하거나 보강하라.

- API 문서를 docs/ 아래로 위치 조정.
- 변경된 파일 목록과 실행 방법(README 영향 있으면 README도 수정)을 함께 커밋 가능한 상태로 만들어라.
